# Make a Plan Plan

## Goal
Create a lightweight, reusable process for producing effective implementation plans in this workspace.

## Current context / assumptions
- The request is planning-only.
- No implementation work should be performed in this turn.
- The user asked for "make a plan plan," which I interpret as a meta-plan for how to create future plans.
- The active deliverable is this markdown file saved under `.hermes/plans/`.

## Proposed approach
Define a short repeatable workflow for turning a user request into a concrete execution plan. The workflow should emphasize:
1. understanding the request,
2. inspecting only the necessary context,
3. identifying impacted files and validation steps,
4. surfacing risks and open questions,
5. producing an actionable plan before any code changes begin.

## Step-by-step plan
1. Clarify the target outcome.
   - Restate the requested end state in one sentence.
   - Identify whether the plan is for a feature, bugfix, refactor, migration, investigation, or operations task.

2. Gather relevant read-only context.
   - Inspect repository structure, relevant docs, and existing code paths.
   - Note any constraints from project conventions, environment, tooling, or user preferences.
   - Avoid implementation or mutating commands during planning.

3. Define scope and assumptions.
   - Separate what is definitely in scope from what is explicitly out of scope.
   - Record assumptions where requirements are implied rather than explicit.
   - Identify any blockers or dependencies that could change the plan.

4. Identify likely files and systems involved.
   - List exact file paths, modules, tests, configs, docs, or services likely to be touched.
   - For non-code tasks, list the relevant systems, interfaces, or operational surfaces.

5. Design the approach.
   - Break the work into phases or milestones.
   - For each phase, describe the intended outcome and why the order makes sense.
   - Include alternate approaches if there is a major tradeoff.

6. Define validation.
   - List the tests, commands, manual checks, or acceptance criteria that would verify success.
   - Include both functional validation and regression checks.

7. Capture risks and open questions.
   - Note uncertainty, integration risk, migration risk, or tooling risk.
   - List the questions that should be answered before execution begins.

8. Produce the final execution-ready plan.
   - Write a concise but concrete markdown plan.
   - Ensure it is actionable enough that implementation can begin without re-discovery.

## Plan template for future use
```md
# <Task title>

## Goal

## Risks / tradeoffs
- ...

- ...
```

## Tests / validation
- Confirm the plan file exists at the requested path.
- Confirm the document includes goal, context, approach, steps, likely files, validation, and risks/open questions.
- Confirm no project files outside the plan were modified.

## Risks / tradeoffs
- A meta-plan may be more abstract than the user intended if they wanted a plan for a specific implementation task.
- Without a concrete target task, file paths and validation steps remain generic.
- Over-standardizing the plan format could make future plans less adaptive to unusual tasks.

## Open questions
- Should this meta-plan become a reusable checklist or template elsewhere in the workspace?
- Does the user want future plans to optimize for brevity or for maximum implementation detail?

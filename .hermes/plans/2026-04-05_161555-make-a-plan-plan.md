# Plan for creating a concrete implementation plan

## Goal
Create a reliable, reusable process for turning a loosely specified request into a concrete execution plan before any implementation work begins.

## Current context / assumptions
- The current request appears to be meta-planning: "make a plan plan."
- No specific feature, bug, or project change has been requested yet.
- This plan therefore focuses on how to produce a high-quality task plan once a target objective is available.
- Planning mode is intended to be non-mutating except for saving this plan file.

## Proposed approach
Use a two-phase planning workflow:
1. Understand the target task and its constraints.
2. Produce an actionable implementation plan with files, validation steps, risks, and open questions.

This keeps planning lightweight when the request is vague, while still producing a plan structure that is immediately useful once the task is clarified.

## Step-by-step plan
1. Identify the target outcome.
   - Determine what needs to be built, changed, fixed, or investigated.
   - Capture the desired end state in one or two sentences.

2. Gather current context.
   - Inspect the relevant repository area, docs, tests, and config.
   - Note existing architecture, conventions, and constraints.
   - Identify whether there are existing skills, prior sessions, or patterns that should be reused.

3. Define assumptions and scope.
   - Separate confirmed facts from assumptions.
   - State what is in scope and what is explicitly out of scope.
   - Call out unknowns that could change the approach.

4. Choose the implementation strategy.
   - Compare the likely approaches briefly.
   - Pick the most practical path based on codebase structure, testability, and risk.
   - Note tradeoffs if there is more than one reasonable option.

5. Break the work into concrete steps.
   - Translate the strategy into an ordered checklist.
   - Keep each step specific enough that it could be executed directly.
   - Group steps by discovery, implementation, testing, and cleanup.

6. List files likely to change.
   - Identify source files, tests, config, docs, and migration points.
   - Include exact paths where possible.
   - Note any files that should be inspected before editing.

7. Define validation.
   - List exact tests, commands, or manual checks needed.
   - Include success criteria for behavior, regressions, and edge cases.
   - Note whether validation is fast, medium, or long-running.

8. Record risks and open questions.
   - Note architectural risks, ambiguous requirements, and dependency issues.
   - Highlight any questions that should be answered before implementation starts.
   - Mark blockers versus non-blocking uncertainties.

9. Save the finished plan.
   - Write the plan into `.hermes/plans/` with a timestamped filename.
   - Ensure the plan is actionable enough for execution without re-discovery.

## Files likely to change
For this meta-plan itself:

For a future task-specific plan, likely categories would include:
- Relevant source files in the active workspace
- Relevant test files under `tests/` or project-specific test directories
- Supporting docs such as `README.md`, `AGENTS.md`, or feature-specific documentation
- Configuration files if the planned work involves tooling, environments, or deployment

## Tests / validation
For the planning process itself:
- Confirm the plan states a clear goal.
- Confirm it includes assumptions, approach, ordered steps, target files, validation, risks, and open questions.
- Confirm the plan is saved under `.hermes/plans/`.
- Confirm the plan does not include implementation work.

For a future task-specific plan, validation should include:
- Exact test targets
- Expected behavioral checks
- Regression coverage
- Any manual QA steps required

## Risks, tradeoffs, and open questions
### Risks
- If the original request is underspecified, the plan may optimize for the wrong outcome.
- Over-planning can waste time for very small tasks.
- Under-planning can miss risky files, hidden dependencies, or validation gaps.

### Tradeoffs
- A short plan is faster to produce but may omit useful implementation detail.
- A detailed plan is more executable but costs more up-front investigation time.
- Early clarification reduces risk but may interrupt momentum.

### Open questions
- What concrete task should this planning workflow be applied to?
- Is the desired output a generic reusable planning template, or a plan for a specific feature/bug?
- Are there project-specific constraints, deadlines, or forbidden areas that future plans should account for?

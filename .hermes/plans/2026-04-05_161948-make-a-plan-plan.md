# Plan: Make a Plan Plan

## Goal
Create a reusable, minimal process for turning a vague request into a concrete implementation plan.

## Current context / assumptions
- The request is intentionally meta: "make a plan plan."
- No specific feature, bug, or project area was provided.
- The desired output for this turn is a planning document only, not implementation.
- This plan should therefore describe how to produce a real execution plan once a concrete task is known.

## Proposed approach
Use a two-phase planning workflow:
1. Establish the task boundaries and success criteria.
2. Convert the clarified task into an actionable implementation plan with files, steps, and validation.

## Step-by-step plan
1. Identify the target task.
   - Determine what is actually being planned: feature work, bugfix, refactor, infra change, research, or migration.
   - Capture the user-visible outcome in one sentence.

2. Gather context.
   - Inspect relevant repo areas, existing docs, tests, and current architecture.
   - Note constraints such as platform, backward compatibility, performance, and deployment requirements.

3. Define success criteria.
   - Write down what must be true when the work is done.
   - Include user-facing behavior, internal correctness, and validation expectations.

4. Map the implementation surface.
   - Identify exact files and subsystems likely to change.
   - Note any config, docs, tests, migrations, or integration points involved.

5. Break the work into ordered steps.
   - Start with investigation/design tasks.
   - Follow with implementation tasks.
   - End with validation, cleanup, and documentation.

6. Specify verification.
   - List exact test targets to run.
   - Include manual verification steps if automated coverage is incomplete.

7. Record risks and open questions.
   - Call out ambiguities, external dependencies, and tradeoffs.
   - Flag anything requiring user confirmation before implementation begins.

## Files likely to change
Because no concrete task was supplied, exact file paths cannot yet be identified.

Once the real task is known, enumerate:
- Source files to modify
- Test files to add/update
- Config or schema files affected
- Documentation or plan files to update

## Tests / validation
For the eventual concrete task, validation should include:
- Targeted unit/integration tests for changed areas
- Existing regression suites relevant to the touched subsystem
- Manual smoke checks for user-facing behavior
- Lint/type/test commands appropriate to the repo

## Risks / tradeoffs
- The main risk is planning too early without a concrete objective.
- Overly generic plans are low-value and may miss critical repo-specific constraints.
- A short clarification step up front usually reduces rework.

## Open questions
1. What actual task should be planned?
2. Is the target a code change, infrastructure change, research task, or process/documentation work?
3. Are there constraints on scope, deadline, or validation depth?

Provide the concrete task to be planned. Once that is known, produce a repo-specific plan with exact file paths, ordered implementation steps, and validation commands.
